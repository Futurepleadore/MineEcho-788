# 安全策略

MineEcho 是本地优先项目，因此很多安全问题会涉及本地运行态数据、凭据、导入文档、AI 应用配置或 Gateway 兼容层状态。默认应将 `.mineecho/`、`.openclaw/` 和 `apps/**/workspace/` 下的内容视为私有数据。

## 支持版本

MineEcho 目前仍处于 1.0 之前。安全修复默认面向当前 `main` 分支，除非后续明确发布维护分支。

## 漏洞报告

如果你发现漏洞，请不要在维护者有时间响应前公开 PoC 或利用细节。

请优先通过邮件私下报告：

- 邮箱：`yk565628110@163.com`

报告中请提供：

- 简洁的问题描述。
- 受影响的 commit、分支或版本。
- 使用占位凭据和非私有样例数据的复现步骤。
- 影响评估：密钥泄露、本地数据暴露、远程代码执行、提示注入、权限绕过或拒绝服务。
- 相关日志，但需要脱敏 token、个人数据、企业数据、文件路径和 session id。

如果邮件暂不可用，可以先开一个最小公开 issue，只说明需要提交私密安全报告，不要包含利用细节或密钥。

## 敏感数据

不要在 issue、PR、截图、日志或复现压缩包中包含：

- API Key、Provider token、Gateway token、企业用户 token、cookie 或认证 header。
- `.env`、`.env.*`、`.mineecho/`、`.openclaw/` 或 `apps/**/workspace/`。
- 导入文档、生成的知识库页面、图谱数据、聊天历史、会议录音或记忆文件。
- 暴露私有基础设施的真实 AI 应用端点。

发布或分享分支前运行：

```sh
npm run check:release
```

如果只想在本地查看诊断结果但不让命令失败：

```sh
node scripts/check-release.mjs --warn-only
```

## 范围

范围内：

- 运行态文件、日志、API 响应或打包产物造成的凭据泄露。
- 不安全的 skill 导入或包解压行为。
- AI 应用连接器泄露配置密钥。
- 知识库导入路径意外暴露本地文件。
- Gateway 兼容层绕过预期认证或隔离。
- 可导致已存私有数据或凭据外泄的提示注入路径。

范围外：

- 需要主动提交恶意本地 `.env` 或运行态数据才成立的攻击。
- 完全发生在 MineEcho 集成边界外的第三方 Provider 问题。
- 仅基于极端本地负载、且缺乏现实缓解价值的拒绝服务报告。

## 维护者响应

预期处理流程：

- 维护者看到报告后进行确认。
- 复现并评估严重程度。
- 准备修复、缓解措施或文档更新。
- 在合适且报告者希望时给予致谢。

不要在回归测试中包含真实密钥或个人/企业私有数据。
